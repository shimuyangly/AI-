# AI求职助手中的 AI 能力应用说明

## 1. 先说明边界：这个项目有没有用大模型

当前线上版本 **没有调用 OpenAI、Claude 或其他大模型 API**。

更准确的定位是：

> 这是一个规则驱动的 AI 产品经理岗位能力分析系统，而不是大模型驱动的生成式 AI 应用。

也就是说，系统不是把 JD 原文直接丢给大模型让它自由总结，而是先把 AI 产品经理岗位中的常见能力拆成结构化规则，再用程序完成识别、映射、统计、打分和建议生成。

当前链路是：

```text
用户粘贴 JD
  |
  v
本地关键词识别
  |
  v
关键词映射到 AI PM 能力域
  |
  v
聚合岗位能力要求
  |
  v
结合用户画像做能力匹配和差距判断
  |
  v
生成 Top关键词、能力要求、能力分析、面试重点、岗位洞察、求职建议
```

## 2. 为什么 MVP 阶段没有直接调用大模型

### 2.1 稳定性

直接调用大模型会更灵活，但同一批 JD 可能每次输出都不一样。对于求职分析工具，用户需要知道结论从哪里来，所以 MVP 先用规则保证结果稳定。

### 2.2 可解释性

本地规则可以解释清楚：

- 哪个词被识别了
- 这个词为什么属于某个能力域
- 哪个画像字段影响了能力判断
- 为什么某项能力被判断为优势、可迁移或短板

这比单纯让大模型输出一段总结更适合早期验证。

### 2.3 成本

一次最多支持 50 个岗位。如果每次都调用大模型，成本会随着用户量和 JD 长度快速增加。当前本地规则运行在浏览器中，几乎没有调用成本。

### 2.4 安全

当前项目是纯前端部署。如果直接在前端调用 AI API，API key 会暴露在浏览器里。后续如果接入大模型，应该通过后端或 Vercel Serverless Function 转发。

## 3. AI 能力识别是怎么做的

### 3.1 关键词规则结构

系统里维护了一份 AI 产品经理岗位关键词字典。每条规则包含 4 个字段：

```ts
{
  keyword: 标准展示词,
  aliases: 同义词 / 别名 / JD 中可能出现的表达,
  category: 关键词类别,
  ability: 映射到的能力域
}
```

例如：

```ts
{
  keyword: 'RAG',
  aliases: ['rag', '检索增强', '知识库', '向量库', '召回', '检索'],
  category: 'AI 应用',
  ability: 'RAG / 知识库'
}
```

这样做的原因是，JD 中不会总是写标准词。例如有的岗位写 RAG，有的岗位写知识库，有的岗位写检索增强，有的岗位写向量库或召回。它们本质上都指向同一类能力。

### 3.2 当前关键词到能力域的对应关系

| 标准关键词 | JD 中可能出现的表达 | 关键词类别 | 映射能力域 |
|---|---|---|---|
| RAG | RAG、检索增强、知识库、向量库、召回、检索 | AI 应用 | RAG / 知识库 |
| Prompt | Prompt、提示词、提示工程、提示词工程 | AI 应用 | Prompt 工程 |
| Agent | Agent、智能体、多智能体、工具调用、function calling | AI 应用 | Agent 产品设计 |
| LLM | LLM、大模型、语言模型、GPT、模型能力 | AI 应用 | AI 应用设计 |
| 问答 | 问答、QA、客服、对话、chatbot、机器人 | 场景 | AI 应用设计 |
| 模型评测 | 评测、测评、效果评估、benchmark、准确率、幻觉、满意度 | 质量 | 模型评测 |
| 数据分析 | 数据分析、数据指标、埋点、漏斗、转化率、留存、看板 | 产品方法 | 数据分析 |
| A/B 测试 | A/B、AB test、AB测试、实验 | 产品方法 | 数据分析 |

| 增长 | 增长、拉新、促活、转化、用户增长 | 业务 | 增长产品 |
| B 端 | B端、B 端、企业、SaaS、客户、后台 | 业务 | B 端产品 |
| C 端 | C端、C 端、用户体验、用户端、App | 业务 | C 端产品 |
| 多模态 | 多模态、图片理解、语音、视频、OCR | AI 应用 | AI 应用设计 |
| 工作流 | 工作流、workflow、自动化、流程编排 | AI 应用 | Agent 产品设计 |

## 4. 识别流程：从 JD 原文到关键词统计

系统会对用户粘贴的每个岗位做文本扫描。

举例，JD 原文里有：

```text
熟悉 LLM、RAG、Agent 基本概念
具备 Prompt Engineering 实战经验
搭建 Agent 效果评估体系，建立测试集并持续优化准确率
```

系统会识别到：

```text
LLM：1 次
RAG：1 次
Agent：2 次
Prompt：1 次
模型评测：2 次
```

然后进入下一步能力映射。

## 5. 映射流程：从关键词到能力要求

关键词本身只说明 JD 出现了哪些词，但求职者真正需要理解的是岗位要求什么能力。

所以系统会把关键词映射为能力域：

```text
RAG、知识库、检索增强、向量库、召回
  -> RAG / 知识库

Prompt、提示词、提示工程
  -> Prompt 工程

Agent、智能体、工作流、工具调用
  -> Agent 产品设计

LLM、大模型、问答、多模态
  -> AI 应用设计

评测、准确率、幻觉、benchmark、测试集
  -> 模型评测
```

映射之后，系统会聚合每个能力域的出现次数，形成“候选人核心能力要求”。

例如：

```text
Agent 产品设计：3 次
模型评测：2 次
Prompt 工程：1 次
RAG / 知识库：1 次
AI 应用设计：1 次
```

这样用户看到的是岗位能力结构，而不是零散词频。

## 6. 能力要求星级是怎么来的

候选人核心能力要求模块会展示星级。

星级不是用户匹配度，而是 **岗位对该能力的要求强度**。

计算思路：

- 先找到本批次中出现次数最高的能力。
- 其他能力按相对出现频次折算为 2-5 星。
- 排名靠后的能力会有星级上限，避免低频能力也显示过高。

示例：

```text
Agent 产品设计出现 6 次，是最高频能力 -> 5 星
模型评测出现 4 次 -> 4 星
Prompt 工程出现 2 次 -> 3 星
```

这个模块回答的是：

> 这批岗位更看重哪些能力？

不是回答：

> 用户和这项能力匹不匹配？

用户匹配关系放在“能力分析”模块里处理。

## 7. 用户画像如何参与分析

用户画像不是只展示，而是会参与能力分析。

当前画像字段包括：

- 工作年限
- 学历背景
- 有无产品经验
- 过往产品类型
- 熟悉的技术
- 擅长的技能

其中对能力分析影响最大的是：

- 熟悉的技术
- 擅长的技能
- 过往产品类型
- 有无产品经验
- 工作年限

### 8.4 有无产品经验 -> 产品基础能力增强

如果用户选择“有产品经验”，系统会对以下能力给予一定基础增强：

- 产品方法论
- 项目推进
- 数据分析
- 商业化理解
- B 端产品
- C 端产品

原因是这些能力在传统产品经理经验中更容易迁移。

如果用户选择“无产品经验”，系统会更倾向于把产品基础能力和 AI 专项能力判断为需要补齐或低置信度迁移。

### 8.5 工作年限 -> 能力成熟度修正

工作年限不会直接决定用户是否具备某项 AI 能力，但会影响能力成熟度判断。

当前逻辑：

- 3-5 年：传统产品能力成熟度更高，项目推进、方法论、商业化等能力更容易被判断为可迁移或优势。
- 1-3 年：有一定迁移基础，但仍需要通过具体案例证明。
- 1 年以内 / 应届生 / 实习生：更偏入门，需要更多 Demo、练习项目或具体作品证明。

## 9. 能力分析是怎么打分的

能力分析模块回答的是：

> 这个用户面对这批 JD，哪些能力是优势，哪些能力可以迁移，哪些能力要重点补齐？

### 9.1 输入

每个能力的判断会读取：

- JD 中该能力的出现次数
- 该能力在 AI PM 岗位中的重要程度
- 用户画像中是否有对应技术、技能、产品类型、经验年限

### 9.2 能力差距优先级公式

当前简化公式：

```text
能力差距优先级 =
JD 提及强度 * 45%
+ AI 岗位关键度 * 30%
+ 用户缺口程度 * 25%
```

解释：

- JD 提及强度：这项能力在本批岗位里出现越多，越重要。
- AI 岗位关键度：RAG、Prompt、Agent、模型评测等 AI 专项能力权重更高。
- 用户缺口程度：用户画像中相关证据越少，缺口越大。

### 9.3 用户画像匹配分

用户画像匹配分来自：

```text
熟悉技术匹配
+ 擅长技能匹配
+ 产品类型匹配
+ 产品经验加成
+ 工作年限成熟度修正
```

示例：

如果 JD 强调 Prompt 工程，用户画像中选择了：

- 熟悉技术：Prompt 工程、Prompt 调优、AI 大模型
- 擅长技能：产品设计、需求分析

那么 Prompt 工程会更容易被判断为“优势能力”或“可迁移能力”。

如果 JD 强调 Agent 产品设计，但用户画像里没有 Agent、工作流、API、项目管理等证据，那么 Agent 产品设计更容易被判断为“重点补齐”。

## 10. 能力标签是怎么判定的

能力分析模块有 3 类标签：

- 优势能力
- 可迁移能力
- 重点补齐

### 10.1 优势能力

判定逻辑：

- 用户画像中有较强直接证据。
- JD 中该能力也有明确要求。

示例：

```text
JD 高频：Prompt、提示词、Prompt Engineering
用户画像：熟悉 Prompt 工程、Prompt 调优、AI 大模型
结果：Prompt 工程 -> 优势能力
```

### 10.2 可迁移能力

判定逻辑：

- 用户画像中没有完全对应的 AI 经验。
- 但有可迁移的产品方法、场景经验或技术基础。

示例：

```text
JD 高频：模型评测、准确率、满意度
用户画像：擅长数据分析、指标体系，熟悉 A/B 测试
结果：模型评测 -> 可迁移能力
```

### 10.3 重点补齐

判定逻辑：

- JD 中能力重要。
- 用户画像中缺少直接证据和迁移证据。

示例：

```text
JD 高频：Agent、工具调用、工作流
用户画像：没有 Agent / 智能体、工作流编排、API、项目管理相关信息
结果：Agent 产品设计 -> 重点补齐
```

## 11. 面试重点是怎么生成的

面试重点不是重新分析 JD，而是基于“能力分析结果”继续生成。

输入：

- 高频能力
- 重点补齐能力
- 用户过往产品类型
- 用户擅长技能
- JD Top关键词

输出分为 4 类：

### 11.1 必备问题

来源：

- JD 中最高频或最关键的能力。

示例：

如果最高频能力是 RAG / 知识库，系统会生成：

```text
如何设计知识库问答链路？
如何处理召回不准、答案无来源和内容更新？
```

如果最高频能力是 Agent 产品设计，系统会生成：

```text
Agent 和普通 Chatbot 的区别是什么？
任务拆解、工具调用、异常兜底怎么设计？
```

### 11.2 案例包装

来源：

- 用户过往产品类型
- 用户擅长技能
- JD 高频方向

示例：

如果用户过往做工具类 / 出行类产品，系统会建议：

```text
先讲原业务里的用户路径和核心痛点；
再讲如果 AI 化，会如何设计输入、输出、兜底和反馈；
最后用产品设计、数据分析等能力证明自己能推进落地。
```

### 11.3 短板补救

来源：

- 能力分析中被标为“重点补齐”的能力。

示例：

如果 Agent 产品设计是重点补齐，系统会建议：

```text
主动承认没有完整 Agent 项目经验；
但说明自己已经理解任务拆解、工具调用、失败重试和人工接管；
最好补一个轻量 Demo 或流程图证明学习进展。
```

### 11.4 临时补课

来源：

- 高频 AI 能力
- 重点补齐能力

示例：

如果要补 RAG：

```text
看懂：切分、Embedding、向量库、召回、重排、溯源；
动手：做一个 20 条资料的问答 Demo；
复盘：记录 5 个答错样本和优化动作。
```

## 12. 一个完整例子

### 12.1 输入 JD

```text
岗位名称：AI 产品经理
薪资：20-35K
岗位职责：
1. 定义 Agent 核心场景，输出 PRD 与优先级
2. 搭建 Agent 效果评估体系，建立测试集并持续优化准确率
3. 推动 AI 工具落地

任职要求：
1. 熟悉 LLM、RAG、Agent 基本概念
2. 具备 Prompt Engineering 实战经验
3. 数据敏感，能用指标证明产品价值
```

### 12.2 关键词识别结果

```text
Agent：多次出现
Prompt：出现
RAG：出现
LLM：出现
模型评测：由“评估体系、测试集、准确率”识别
数据分析：由“指标、数据敏感”识别
项目推进：由“推动落地”识别
```

### 12.3 能力映射结果

```text
Agent -> Agent 产品设计
Prompt -> Prompt 工程
RAG -> RAG / 知识库
LLM -> AI 应用设计
评估体系 / 测试集 / 准确率 -> 模型评测
指标 / 数据敏感 -> 数据分析
推动落地 -> 项目推进
```

### 12.4 结合用户画像

如果用户画像是：

```text
工作年限：1-3 年
有产品经验：有
过往产品类型：工具类、出行 / 交通
熟悉技术：AI 大模型、RAG / 知识库、Prompt 调优

```

可能得到：

```text
RAG / 知识库：优势能力或可迁移能力
Prompt 工程：优势能力或可迁移能力

Agent 产品设计：重点补齐
模型评测：可迁移能力
项目推进：可迁移能力
```

### 12.5 输出面试重点

```text
必备问题：
Agent 和普通 Chatbot 的区别是什么？
如何设计知识库问答链路？

案例包装：
把工具类产品经验包装成 AI 工具任务流设计。

短板补救：
Agent 项目经验不足时，用任务拆解、工具调用、失败兜底流程图补证明。

临时补课：
补 Agent 工作流、RAG 链路和模型评测指标。
```

## 13. 当前 AI 能力应用的价值

这个项目的 AI 能力价值不在于调用了大模型，而在于把 AI 产品经理岗位知识结构化了。

具体价值：

- 把复杂 JD 转成可读的能力结构。
- 把零散 AI 词汇归类到产品能力域。
- 把用户画像和岗位要求做可解释匹配。
- 把能力差距转成面试准备事项。
- 为后续大模型增强提供稳定的结构化输入。

## 14. 后续如果接入大模型，应该怎么做

推荐方案是“规则分析 + 大模型生成”，而不是完全依赖大模型。

```text
JD 原文
  |
  v
本地规则分析
  |
  | 输出结构化事实：
  | - Top关键词
  | - 能力要求
  | - 用户画像
  | - 能力差距
  | - 面试重点草稿
  v
后端 API
  |
  v
大模型
  |
  v
生成更自然的：
  - 能力差距说明
  - 简历改写
  - 面试回答话术
  - 作品集建议
```

为什么这样设计：

- 规则负责稳定和可解释。
- 大模型负责表达、总结和个性化话术。
- 后端负责保护 API key、限流和成本控制。

## 15. 面试中可以这样介绍

可以这样讲：

> 这个项目当前没有直接调用大模型 API。我在 MVP 阶段选择先做规则驱动的 AI 岗位能力分析，因为求职分析工具需要稳定和可解释。我把 AI 产品经理岗位中常见的 RAG、Prompt、Agent、LLM、模型评测等要求拆成关键词字典和能力域映射。系统会先识别 JD 中的关键词，再映射到能力要求，然后结合用户画像中的工作年限、产品经验、过往产品类型、熟悉技术和擅长技能，判断哪些能力是优势、哪些可以迁移、哪些需要重点补齐。  
>
> 这样做的价值是，用户不只是看到“岗位提到了 Agent”，而是能看到“Agent 产品设计是本批岗位的核心能力，而我当前画像中缺少工作流和工具调用经验，所以需要重点补齐”。后续如果接入大模型，我会把这些结构化结果传给大模型，让它生成更自然的简历改写、面试话术和作品集建议，而不是让大模型直接从原始 JD 自由发挥。

## 16. 面试中需要避免的说法

不建议说：

- 我做了一个大模型应用。
- 我实现了 RAG / Agent 链路。
- 系统用 AI 自动理解 JD。
- 系统调用大模型生成分析。

更准确的说法：

- 我做了一个 AI 产品经理岗位能力分析工具。
- 当前版本是规则驱动，没有调用大模型 API。
- 我把 AI PM 岗位能力做了结构化建模。
- 系统通过关键词识别、能力映射和画像打分生成分析结果。
- 后续可以接入大模型做表达增强和个性化话术生成。
